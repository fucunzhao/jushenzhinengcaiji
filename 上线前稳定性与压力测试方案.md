# 上线前稳定性与压力测试方案

## 1. 测试目标

本系统上线前重点验证四件事：

1. 前端页面稳定：不同角色登录后能看到正确看板，常用模块可展开折叠，任务库筛选、任务详情、正式任务登记、道具比对、设备选择等功能不卡死。
2. 后台接口稳定：登录、人员库、任务分配、任务领取锁、基础库上传下载、作业开始/完成接口返回正确。
3. 数据库稳定：`data/store.json` 中人员、任务分配、任务领取锁、基础库覆盖数据结构完整，不出现重复账号、采集员无培训师、重复任务锁等问题。
4. 并发压力可承受：多个培训师同时查看任务库、复制正式登记、分配任务、采集员开始/完成任务时，系统不崩溃、不重复领取、不丢数据。

## 2. 前端稳定性测试

### 2.1 登录与角色看板

测试账号：

- 企业负责人：`fucunzhao / zzdh6886`
- 项目经理、培训师、采集员：由负责人账号在人员库中开通后测试

检查项：

- 企业负责人：能看到全员状态、人员库、任务库、道具库、场地库、基础库维护中心。
- 项目经理：能看到日报填写、培训师截止检查、任务进度表、基础库维护中心。
- 培训师：能看到查找并分配任务、正式任务登记/批量申请、道具准备、采集员进度。
- 采集员：能看到自己的任务、道具清单、采集步骤、设备下拉、开始/完成填报。

### 2.2 高频操作流程

培训师流程：

1. 搜索任务。
2. 点击任务卡片查看详情。
3. 进入“正式任务登记 / 批量申请”。
4. 生成 `0719_柳州_1/2/3` 后缀任务。
5. 复制正式登记表行。
6. 分配不同采集员、场地、房间。
7. 复制领取提醒。
8. 确认其他培训师账号中该任务显示“今日已领取”，无法重复领取。

采集员流程：

1. 登录采集员账号。
2. 按日期查看任务。
3. 核对完整任务名和后缀。
4. 查看道具清单和步骤。
5. 下拉选择设备号。
6. 点击开始作业。
7. 完成后填写有效时长、是否全部完成、完成条数、剩余条数。
8. 培训师、项目经理、负责人看板同步显示。

### 2.3 基础库维护流程

负责人或项目经理：

1. 下载当前库 Excel。
2. 修改人员库、任务库、道具库、场地库、设备库。
3. 上传修改后的 Excel。
4. 确认页面任务库、道具库、场地库、人员库刷新。
5. 确认新增采集员必须绑定培训师。
6. 确认上传错误格式时有明确提示。

## 3. 后台数据库测试

检查 `data/store.json`：

- `users` 必须存在，且至少有一个企业负责人。
- `username` 不允许重复。
- `phone` 不允许重复。
- 采集员必须有有效 `trainerId`。
- `assignments` 中的采集员、培训师必须存在。
- `taskClaims` 中同一日期、同一基础任务只能有一个有效培训师领取。
- `libraries` 可以为空，但一旦上传后必须包含结构化基础库数据。

## 4. 接口冒烟测试

必须通过：

- `GET /`
- `POST /api/login`
- `GET /api/me`
- `GET /api/users`
- `GET /api/assignments`
- `GET /api/task-claims`
- `GET /api/libraries`

建议在每次修改代码后执行一次冒烟测试。

## 5. 压力测试设计

轻量压力测试目标：

- 并发用户数：20、50、100 三档。
- 持续时间：30 秒、60 秒、180 秒三档。
- 请求组合：
  - 40% 查看任务库页面。
  - 20% 登录。
  - 15% 拉取人员库。
  - 15% 拉取任务分配。
  - 5% 拉取任务领取锁。
  - 5% 拉取基础库。

合格标准：

- 错误率低于 1%。
- 平均响应小于 300ms。
- P95 响应小于 1000ms。
- 服务不中断。
- `data/store.json` 不损坏。

## 6. 上线前通过标准

上线前至少满足：

- 冒烟测试全部通过。
- 数据库体检无错误。
- 企业负责人、项目经理、培训师、采集员四类账号各完成一次完整流程。

- 压力测试 50 并发 60 秒无崩溃。

## 7. 风险提醒

当前系统使用本地 JSON 文件作为后台数据库，适合小团队本地内网使用。若后续多人高并发、跨地点访问、长期保存大量记录，建议升级为 SQLite、PostgreSQL 或 MySQL，并增加自动备份、操作日志和数据恢复机制。
